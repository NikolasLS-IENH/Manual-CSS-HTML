Interação e Formulários

Aprenda a criar formulários HTML acessíveis, validados automaticamente pelo navegador e com experiência de usuário excepcional, usando apenas HTML e CSS.

Este guia cobre tudo que você precisa saber para implementar formulários profissionais em seu projeto, desde a estrutura semântica até validações nativas e feedback visual.

Introdução

Formulários são um dos componentes mais fundamentais e vitais da web. Eles permitem que usuários interajam com suas aplicações: criando contas, fazendo compras, enviando mensagens ou qualquer troca de dados. A qualidade de um formulário impacta diretamente a taxa de conversão e satisfação do usuário.

Este guia define o padrão oficial de formulários do projeto, cobrindo acessibilidade, validação e experiência do usuário. O ponto mais importante: você não precisa de JavaScript para criar formulários funcionais. Com apenas HTML + CSS é possível validar dados automaticamente, fornecer feedback visual e melhorar significativamente a experiência do usuário.

JavaScript é um complemento opcional para funcionalidades avançadas — não o ponto de partida.

Objetivos

Este guia estabelece os pilares para que todos os formulários do projeto sejam consistentes, acessíveis e usáveis. Você aprenderá a:

  • Estrutura de formulários — HTML semântico com <form>, <label> e estrutura clara
  • Tipos de campos — Usar o input correto para cada tipo de dado (email, número, data, etc.)
  • Validações nativas — Aproveitar validação automática do navegador (required, minlength, pattern, etc.)
  • Estilos visuais — CSS para feedback visual imediato (:valid, :invalid, :focus)
  • Acessibilidade — Garantir que todos os usuários, incluindo aqueles com deficiências, consigam preencher formulários

O que são formulários?

Um formulário HTML é um conjunto de campos que coleta dados do usuário e os envia para o servidor. Pode ser desde um simples login até um complexo formulário de cadastro com múltiplos passos.

Componentes principais:

  • <form> — O contêiner semântico que agrupa campos relacionados
  • <input>, <select>, <textarea> — Campos para entrada de dados
  • <label> — Rótulos descritivos para cada campo (essencial para acessibilidade)
  • <button> — Acionador para submissão ou reset

O ciclo de um formulário:

  1. Usuário vê os campos com rótulos claros
  2. Preenche os dados solicitados
  3. Clica o botão "Enviar"
  4. O navegador valida os dados automaticamente (validação nativa)
  5. Se válido, dados são enviados ao servidor
  6. Se inválido, mensagens de erro aparecem antes de enviar

Ponto chave: Este guia cobre a implementação HTML/CSS. O envio dos dados ao servidor requer backend (não coberto aqui).

Estrutura base (OBRIGATÓRIO)

Todo formulário deve ter uma estrutura clara e previsível. Isso ajuda navegadores, leitores de tela e usuários a entender como preencher e enviar o formulário.

Aqui está o padrão mínimo obrigatório que deve ser seguido em todos os formulários do projeto:

<form class="form">
    <div class="form-group">
        <label for="nome">Nome</label>
        <input id="nome" type="text" required>
    </div>
    <button type="submit">Enviar</button>
</form>

Por que cada elemento importa

<form>
O contêiner semântico que encapsula todo o formulário. Diferente de uma div, indica explicitamente aos navegadores que se trata de um formulário, habilitando funcionalidades nativas como submissão e validação.
.form-group
Agrupa logicamente um rótulo com seu campo. Essa organização facilita a estilização em CSS e manutenção do código. Cada "grupo" é uma unidade independente.
<label>
Descreve o propósito do campo de forma acessível. O atributo for vincula o rótulo ao campo via id. Usuários de tecnologias assistivas (leitores de tela) dependem disso.
<input>
Campo de entrada com tipo (text, email, etc.) que define seu comportamento. O atributo id deve corresponder ao for do rótulo. required marca o campo como obrigatório.
<button type="submit">
Botão que envia o formulário. O tipo submit é crucial — apenas botões desse tipo acionam a submissão automática quando o usuário pressiona Enter.

Nota importante: Nunca use <input type="button"> ou uma <div> para submissão. Isso quebra acessibilidade e validação automática do navegador.

Por que usar <label>?

O label significa "rótulo". Labels são frequentemente negligenciados e substituídos apenas por placeholders. Isso é um erro. Eles são essenciais para acessibilidade e usabilidade.

Benefícios principais:

  • Acessibilidade: Leitores de tela anunciam o rótulo quando o campo recebe foco, informando ao usuário cego o que deve ser preenchido.
  • Usabilidade: Clicar no rótulo move o foco para o campo, aumentando significativamente a área clicável — especialmente importante em mobile.
  • Contexto permanente: O placeholder desaparece ao digitar. O label fica sempre visível.

Comparação: Sem vs. Com label

Sem label (apenas placeholder):

<input type="email" placeholder="seu@email.com">

Problemas: Usuários com leitores de tela ouvem só "campo de texto". Placeholder desaparece ao digitar. Sem dica sobre o campo.

Com label (recomendado):

<label for="email">Email</label>
<input id="email" type="email" placeholder="seu@email.com">

Vantagens: Claro para todos. Leitor de tela anuncia "Email, campo de email". Clicar no label ativa o input. Maior área clicável.

Dica importante: Placeholder é uma dica, não um rótulo. Use ambos para experiência completa.

Tipos de <input>

O elemento <input> muda de comportamento de acordo com o atributo type. Use o tipo certo para cada dado.

Texto text

<input type="text" placeholder="Seu nome">

Usado para:

  • Nome
  • Cidade

Email email

<input type="email" required>

O que acontece:

  • O navegador valida automaticamente o formato.
  • Exige @ e domínio válido.

    • O que acontece:

    • Os caracteres são escondidos.
    • Pode exigir tamanho mínimo.

    Número number

    <input type="number" min="0" max="10">

    Senha password

    <input type="password" minlength="6">

    O que acontece:

    • Os caracteres são escondidos.
    • Pode exigir tamanho mínimo.

    Número number

    <input type="number" min="0" max="10">

    Radio radio

    <input type="radio" name="plano" value="basic"> Básico
<input type="radio" name="plano" value="pro"> Pro

    Importante:

    • Mesmo name → apenas uma escolha é permitida.

    Checkbox checkbox

    <input type="checkbox"> Aceito os termos

    Permite:

    • Múltiplas seleções (vários checkboxes juntos).
    • Confirmações (ex.: aceite de termos).

Select (lista suspensa)

Use <select> quando há muitas opções e espaço limitado na tela. É mais compacto que radios e mais permitido que inputs de texto.

<label for="pais">País</label>
<select id="pais" required>
    <option value="">Selecione um país</option>
    <option value="br">Brasil</option>
    <option value="pt">Portugal</option>
    <option value="us">Estados Unidos</option>
</select>

Quando usar select:

  • Mais de 5 opções (se menos, use radios)
  • Espaço limitado na tela
  • Opções predefinidas (não digitáveis)

Dica importante:

A primeira <option> com value="" funciona como placeholder. Força o usuário a escolher explicitamente uma opção real (evita seleção acidental da primeira).

Textarea (texto longo)

Para texto longo com múltiplas linhas use <textarea>, nunca <input type="text">. Textarea permite quebras de linha e redimensionamento. 

<label for="mensagem">Sua mensagem</label>
<textarea id="mensagem" rows="5" placeholder="Digite aqui..."></textarea>

Atributos úteis:

  • rows — Altura inicial em linhas (padrão: 2)
  • cols — Largura (raro, use CSS em vez)
  • placeholder — Dica de preenchimento
  • maxlength — Número máximo de caracteres
  • required — Marca como obrigatório

Casos de uso:

  • Comentários
  • Mensagens ou feedback
  • Descrições ou bios
  • Qualquer texto livre de múltiplas linhas

Botões

Existem três tipos de botões em formulários, cada um com um propósito diferente e comportamento distinto.

type="submit" — Enviar

<button type="submit">Enviar</button>

O que faz: Ativa a validação automática do navegador e dispara o evento submit. Permite submissão ao pressionar Enter.

type="reset" — Limpar (raro)

<button type="reset">Limpar</button>

O que faz: Reseta todos os campos aos valores iniciais. Raramente útil. O usuário basicamente, perde todo seu trabalho. Não recomendado.

type="button" — Ação customizada

<button type="button">Cancelar</button>

O que faz: Não submete o formulário. Requer JavaScript para funcionar. Use para ações como "Cancelar", "Limpar manualmente" ou abrir modais.

Regra importante: Use type="submit" para enviar formulários. Use type="button" apenas quando a ação não envolve submissão de dados.

Validações nativas (SEM JavaScript!)

HTML5 oferece validação automática no navegador. Você não precisa escrever uma linha de JavaScript para validação básica. O navegador faz o trabalho pesado.

required — Campo obrigatório

<input type="text" required>

Se o usuário tenta enviar o formulário vazio, o navegador bloqueia e mostra: "Por favor, preencha este campo".

minlength e maxlength — Comprimento de texto

<input type="text" minlength="3" maxlength="50">

Define comprimento mínimo e máximo. Se o usuário digita menos de 3 caracteres, o navegador bloqueia o envio.

min e max — Intervalo numérico

<input type="number" min="0" max="100">
<input type="date" min="2024-01-01" max="2024-12-31">

Define limite mínimo e máximo. Funciona com number, date, range e time.

pattern — Expressão regular (regex)

<input type="text" pattern="[A-Za-z]+" title="Apenas letras">

Valida contra um padrão regex customizado. Exemplo: [A-Za-z]+ = apenas letras, \d{3}-\d{4} = telefone (123-4567), etc.

Validação automática por type

<input type="email" required>
<input type="url" required>
<input type="date" required>

O navegador valida formato automaticamente. Email requer @. URL requer http://. Cada tipo tem suas próprias regras.

Como funciona o ciclo:

  1. Usuário clica "Enviar"
  2. Navegador valida cada campo contra suas regras
  3. Se algum inválido → mostra mensagem de erro e para
  4. Se todos válidos → dispara evento submit (você envia ao servidor aqui)

Importante: Validação no navegador é apenas para UX. Sempre valide no servidor também — um usuário poderia hackear DevTools e contornar validação!

Feedback visual com CSS (:valid, :invalid, :focus)

CSS permite mostradores visuais imediatos sobre o estado de validação dos campos. Isso mostra ao usuário imediatamente se algo está certo ou errado, antes de enviar.

:valid — Campo válido

input:valid {
  border: 2px solid green;
  background-color: rgba(0, 255, 0, 0.05);
}

Aplicado quando o campo passou em todas as validações (type, required, minlength, pattern, etc.).

:invalid — Campo inválido

input:invalid {
  border: 2px solid red;
  background-color: rgba(255, 0, 0, 0.05);
}

Aplicado quando o campo violou alguma validação.

:focus — Campo em foco

input:focus {
  outline: none;
  border-color: blue;
  box-shadow: 0 0 0 3px rgba(0, 0, 255, 0.1);
}

Mostra qual campo o usuário está preenchendo. Essential para acessibilidade — usuários que navegam apenas por teclado precisam saber qual campo está ativo.

Cuidado: :invalid em tempo real

Por padrão, :invalid ativa assim que o campo é inválido, mesmo enquanto o usuário ainda está digitando. Isso é frustrante:

/* Ruim: mostra erro enquanto digitando */
input:invalid { border: 2px solid red; }

/* Melhor: mostra erro apenas após sair do campo */
input:invalid:not(:focus) { border: 2px solid red; }

Exemplo completo recomendado:

input,
textarea,
select {
  padding: 12px;
  border: 1px solid #ccc;
  border-radius: 4px;
  font-size: 16px;
}

input:focus,
textarea:focus,
select:focus {
  outline: none;
  border-color: blue;
  box-shadow: 0 0 0 3px rgba(0, 0, 255, 0.1);
}

input:valid:not(:focus),
textarea:valid:not(:focus) {
  border-color: green;
}

input:invalid:not(:focus),
textarea:invalid:not(:focus) {
  border-color: red;
}

Por que feedback visual importa:

  • Usuário entende erros imediatamente, sem esperar envio
  • Reduz taxa de reenvio incorreto
  • Melhora confiança e satisfação do usuário
  • Acessível para usuários com deficiências visuais (com cores + ícones/texto)

Padrão visual recomendado

Aqui estão padrões CSS recomendados para estruturar formulários de forma consistente e profissional em todo o projeto.

Estrutura CSS básica:

.form {
  max-width: 400px;
  margin: 0 auto;
}

.form-group {
  display: flex;
  flex-direction: column;
  margin-bottom: 1.5rem;
  gap: 0.5rem;
}

label {
  font-weight: 500;
  font-size: 14px;
  color: #333;
}

input,
textarea,
select {
  padding: 12px;
  border: 1px solid #ccc;
  border-radius: 4px;
  font-size: 16px;
  font-family: inherit;
}

Estados visuais (com feedback):

input:focus,
textarea:focus,
select:focus {
  outline: none;
  border-color: #0066cc;
  box-shadow: 0 0 0 3px rgba(0, 102, 204, 0.1);
}

input:valid:not(:focus),
textarea:valid:not(:focus) {
  border-color: #28a745;
}

input:invalid:not(:focus),
textarea:invalid:not(:focus) {
  border-color: #dc3545;
}

button {
  padding: 12px 24px;
  font-size: 16px;
  font-weight: 500;
  border: none;
  border-radius: 4px;
  cursor: pointer;
  background-color: #0066cc;
  color: white;
}

button:hover {
  background-color: #0052a3;
}

Dica importante para mobile:

input,
textarea,
select {
  font-size: 16px; /* Previne zoom ao focar em iOS */
}

Erros comuns (evite isso)

Aqui estão os erros mais frequentes que degradam acessibilidade, validação e experiência do usuário:

Não usar <label>

Problema: Usuários cegos com leitores de tela não sabem qual é o campo. Área clicável muito pequena.

Usar apenas placeholder como rótulo

Problema: Placeholder desaparece ao digitar. Usuário perde contexto. Não passa em validação de acessibilidade.

Usar <div> com onclick para "submit"

Problema: Quebra validação automática do navegador. Não funciona ao pressionar Enter. Falha em acessibilidade (tabindex, roles).

Usar type="button" quando deveria ser type="submit"

Problema: Evita ativação de validação automática. Usuário pensa que nada aconteceu.

Usar type="text" para tudo

Problema: Navegadores não conseguem oferecer teclados especializados em mobile. Experiência degradada.

Remover :focus outline apenas para "deixar bonito"

Problema: Usuários que navegam apenas por teclado não conseguem saber qual campo está ativo. Falha crítica em acessibilidade. Sempre substitua com box-shadow ou border-color.

Confiar apenas em validação do navegador

Problema: Um usuário poderia abrir DevTools e hackear a validação. Sempre valide os dados no servidor também.

Boas práticas

Siga estas recomendações para criar formulários de alta qualidade, acessíveis e fáceis de usar:

HTML e estrutura:

  • Sempre use <form>, <label> e <button type="submit">
  • Use o tipo de input correto (email, number, date, etc.) — não type="text" para tudo
  • Vinculação correta: for em label com id em input
  • Use required para campos obrigatórios

Validação:

  • Use validações nativas do HTML (minlength, pattern, min/max, etc.)
  • Sempre valide no servidor também — nunca confie apenas no navegador
  • Forneça mensagens de erro claras e úteis

CSS e feedback:

  • Dê feedback visual com :valid e :invalid (mas não em tempo real)
  • Mantenha :focus outline ou substitua por box-shadow/border-color
  • Mantenha fonte em 16px mínimo em mobile (previne zoom)
  • Mantenha padrões visuais consistentes em todo o projeto

Acessibilidade:

  • Sempre use labels (essencial para leitores de tela)
  • Não remova :focus outline sem alternativo
  • Use placeholder como dica, não como rótulo
  • Teste com leitores de tela (NVDA, JAWS, VoiceOver)
  • Teste navegação apenas com teclado (Tab, Enter, Shift+Tab)

UX:

  • Mantenha formulários simples e focados
  • Agrupe campos relacionados com fieldset
  • Use placeholder para exemplos, não instruções
  • Teste em desktop, tablet e mobile

Exemplo completo explicado

Um formulário de login real, com todas as boas práticas aplicadas. Este é o padrão que você deve seguir em todos os formulários do projeto.

<form class="form">
  <div class="form-group">
    <label for="email">Email</label>
    <input 
        id="email" 
        type="email" 
        placeholder="seu@email.com"
        required
    >
  </div>

  <div class="form-group">
    <label for="password">Senha</label>
    <input 
        id="password" 
        type="password" 
        minlength="8"
        required
    >
  </div>

  <div class="form-group">
    <label>
      <input type="checkbox" required>
      Lembrar de mim
    </label>
  </div>

  <button type="submit">Entrar</button>
</form>

O que está correto aqui:

  • <form> semântico (não div ou article)
  • Email com validação automática (@ obrigatório)
  • Password com minlength=8 (força mínima)
  • Checkbox com label (pode clicar no texto)
  • type="submit" (ativa validação nativa)
  • Todos os campos com labels acessíveis
  • Estrutura clara e previsível
  • Pronto para estilização CSS

O que acontece quando o usuário interage:

  1. Usuário vê "Email" e sabe o que preencher
  2. Clica no campo, vê outline/shadow (feedback de foco)
  3. Digita um email inválido (ex: "teste"), vê borda vermelha
  4. Corrige para "teste@email.com", vê borda verde
  5. Senhas menos de 8 caracteres, navegador bloqueia
  6. Marca checkbox, clica "Entrar"
  7. Navegador valida, envia ao servidor

Conclusão

Você aprendeu a criar formulários com HTML semântico, validação nativa e feedback visual em CSS. Isso é mais do que 90% dos desenvolvedores web conseguem fazer bem.

Recapitulando, com apenas HTML + CSS você consegue:

  • Criar formulários semanticamente corretos
  • Validar dados automaticamente (sem JavaScript)
  • Fornecer feedback visual imediato
  • Garantir acessibilidade para usuários com deficiências
  • Otimizar para mobile com teclados corretos
  • Melhorar significativamente a experiência do usuário
